# Design — tui-welcome-skeleton

## TL;DR

Ten files (7 new + 3 modified) wire OpenTUI into verbum at the outermost presentation ring, behind a three-line argv branch in `src/index.tsx`. The welcome screen is a pure view — zero inward arrows into domain/application/api — built from a hardcoded `welcome-content.ts`, a generated `banner.ts`, a `<WelcomeScreen>` JSX component, and a `useReducer`-driven state machine whose `welcomeReducer` returns `[state, Effect | null]` exactly as Bubble Tea's `Update(msg) → (model, cmd)` signature requires. `tuiDriver(): Promise<void>` owns the OpenTUI renderer handle, runs the effect loop, and resolves on `Effect.Quit` so `src/index.tsx` can `await` it and call `process.exit(0)` — symmetric with the existing CLI path.

## Architecture overview

The slice touches three concentric zones:

1. **Entry (`src/index.tsx`)** — already the wiring layer. Adds an argv branch that chooses TUI vs. existing CLI driver. No business logic, no parsing. This is the only file modified inside `src/`.
2. **Presentation — TUI (`src/tui/`)** — new directory at the same hexagonal ring as `src/cli/`. Holds the renderer driver, the welcome screen component, and the welcome reducer. Mirrors how `src/cli/run.ts` + `src/cli/render.ts` sit at the outer ring for CLI presentation.
3. **Presentation — shared with CLI (`src/cli/banner.ts`, `src/cli/welcome-content.ts`)** — the wordmark and verse constants live in `src/cli/` because they are also reused by future `--version` / `--help` CLI output (per `docs/ui-sketches.md`). Putting them under `src/tui/` would force a CLI-side import to cross into `src/tui/` later, which inverts the layering. Keeping them in `src/cli/` keeps `src/tui/` purely about the TUI runtime.

> **Welcome screen has NO inward arrows — it bypasses domain/application/api by design.** This is intentional and hexagonally valid: a presentation view with zero domain dependencies is a permitted outermost-ring composition (decoration, not workflow). The verses are content constants, not data fetched through a port. If the welcome screen ever needed live verse lookup, that would be a different slice, gated behind the `BibleRepository` port and offline-fallback work.

The `<WelcomeScreen>` is a JSX composition of three regions stacked vertically:

```
┌───────────────── <WelcomeScreen> ─────────────────┐
│  <Banner />              ← banner.ts string       │
│  <BookFrame />           ← welcome-content.ts     │
│      ✦ Genesis 1:1   |   ✦ John 3:16              │
│              v0.1.0                               │
│  <StatusLine />          ← "? help • q quit"      │
└───────────────────────────────────────────────────┘
```

The book-frame region renders the locked `docs/ui-sketches.md` layout: angled `╱`/`╲` edges, two pages of scripture text, version literal between bottom edges, drop-shadow `░░`. The verse text inside the pages has **no inner border** (per `house-rules.md` rule from `docs/ui-sketches.md`: "verses must breathe — never decorate scripture"). The book-frame *is* decoration around the page; the verse text inside is plain.

## Types

All TypeScript signatures for the new code:

```ts
// ─────────────────────────────────────────────────────────────────────────
// src/cli/banner.ts  (generated, committed)
// ─────────────────────────────────────────────────────────────────────────

// The Isometric1 figlet wordmark for "verbum".
// Generated by `bun run generate:banner`. Do NOT edit by hand —
// re-run the script to regenerate.
export const BANNER: string;
//   The string is a multi-line template literal containing the exact
//   output of `bunx figlet -f Isometric1 verbum`, with no leading/trailing
//   newlines stripped. Embedded as a single `const`, no array, no joins.

// ─────────────────────────────────────────────────────────────────────────
// src/cli/welcome-content.ts
// ─────────────────────────────────────────────────────────────────────────

// Genesis 1:1 (BSB) — left page of the book-frame. Hardcoded string
// constant, never fetched. See design D1.
export const GENESIS_1_1_TEXT: string;

// John 3:16 (BSB) — right page of the book-frame. Hardcoded string
// constant, never fetched.
export const JOHN_3_16_TEXT: string;

// Version literal rendered between the bottom edges of the book-frame.
// Sourced from package.json version at write time, NOT read at runtime
// (avoids a runtime dependency on package.json layout).
export const WELCOME_VERSION: string;

// ─────────────────────────────────────────────────────────────────────────
// src/tui/welcome/welcome-reducer.ts
// ─────────────────────────────────────────────────────────────────────────

// WelcomeState — the reducer's model. v1 has a single state because the
// welcome screen has no internal transitions (no loading, no overlay).
// Kept as a discriminated union with one variant so future states (e.g.
// "exiting") plug in without changing the reducer signature.
export type WelcomeState =
  | { kind: "active" };

// WelcomeAction — events that have happened. Past-tense per Rule 10.
// `KeyPressed` is the only action this slice needs.
export type WelcomeAction =
  | { type: "KeyPressed"; key: string };

// Effect — descriptors of side effects the runner should execute.
// `Quit` is the only effect this slice needs. We use `null` (not
// `Effect.None`) for "no effect" to match the existing dialect — the
// reducer in `house-rules.md` Rule 8 returns `[State, Effect | null]`,
// and the apply phase of v1-architecture-spike already established
// `null` as the no-effect convention for this codebase. No `None` variant.
export type Effect =
  | { kind: "quit" };

// welcomeReducer — pure function. Given current state + action,
// returns the next state and an optional effect descriptor.
// MUST NOT touch IO. MUST NOT call any use case.
//
// Bubble Tea port: `func (m welcomeModel) Update(msg tea.Msg) (tea.Model, tea.Cmd)`.
export function welcomeReducer(
  state: WelcomeState,
  action: WelcomeAction,
): [WelcomeState, Effect | null];

// initialState — exported for the React component's `useReducer` seed
// and for the tests. Plain object literal, no factory function.
export const initialWelcomeState: WelcomeState;

// ─────────────────────────────────────────────────────────────────────────
// src/tui/welcome/welcome-screen.tsx
// ─────────────────────────────────────────────────────────────────────────

// Props — externally-controlled bridge so the component is testable in
// isolation if we ever want to. The driver passes a `dispatch` from
// `useReducer` plus the current state. The component does NOT call
// `useReducer` itself — the driver owns the reducer hookup so the
// effect runner can listen on every dispatch.
export type WelcomeScreenProps = {
  state: WelcomeState;
  dispatch: (action: WelcomeAction) => void;
};

// WelcomeScreen — pure JSX composition. No `useEffect`, no `useState`
// for business state (Rule 8/9). Renders banner + book-frame + status line.
export function WelcomeScreen(props: WelcomeScreenProps): JSX.Element;

// ─────────────────────────────────────────────────────────────────────────
// src/tui/tui-driver.ts
// ─────────────────────────────────────────────────────────────────────────

// tuiDriver — initialises the OpenTUI renderer, mounts <WelcomeScreen>,
// wires the keypress dispatch, runs the effect loop, and resolves the
// returned Promise once the reducer emits Effect.quit.
//
// The Promise resolves on `Effect.quit`. It does NOT call process.exit —
// that is the entry point's job (D4, symmetric with src/cli/run.ts).
export function tuiDriver(): Promise<void>;

// runEffect — internal helper inside tui-driver.ts (NOT exported).
// Given an effect descriptor and the renderer handle, executes the
// effect. For `quit`: calls renderer.destroy() and signals the outer
// Promise to resolve. Pure switch on `effect.kind`, exhaustive `never`
// fallthrough.
//
// Signature in the file (private):
//   function runEffect(
//     effect: Effect,
//     renderer: CliRenderer,
//     resolve: () => void,
//   ): void;

// ─────────────────────────────────────────────────────────────────────────
// src/index.tsx — argv branch contract
// ─────────────────────────────────────────────────────────────────────────

// The exact branch condition lives in src/index.tsx:
//
//   const argv = Bun.argv.slice(2);
//
//   if (argv.length === 0) {
//     await tuiDriver();
//     process.exit(0);
//   }
//
//   const exitCode = await run(argv);
//   process.exit(exitCode);
//
// Condition: `argv.length === 0` means the user invoked `verbum` with NO
// positional arguments and NO flags. Anything else (one or more tokens of
// any kind, including future `--version` / `--help`) falls through to the
// existing `run(argv)` CLI dispatch. No special-casing of flags here —
// that's a future slice's concern. SCN-5 from v1 changes from "stderr +
// exit 2" to "TUI welcome + exit 0".
```

## File shapes

### NEW — `src/cli/banner.ts`

- **Path**: `src/cli/banner.ts`
- **Exports**: `BANNER: string`
- **Purpose**: Pre-rendered Isometric1 figlet wordmark for "verbum", embedded as a single multi-line template literal. This file is **generated** by `bun run generate:banner` and committed. The runtime never invokes figlet. Contributors regenerate by running the script when the wordmark needs to change (font swap, brand tweak).
- **Does NOT**: import figlet at runtime; perform any computation; depend on the filesystem; export anything but `BANNER`. This is a leaf module.
- **Why in `src/cli/`** (not `src/tui/`): the wordmark is also used by future `--version` and `--help` CLI output per `docs/ui-sketches.md`. Co-locating with `welcome-content.ts` keeps both shared-presentation constants in one place.

### NEW — `src/cli/welcome-content.ts`

- **Path**: `src/cli/welcome-content.ts`
- **Exports**: `GENESIS_1_1_TEXT`, `JOHN_3_16_TEXT`, `WELCOME_VERSION`
- **Purpose**: Hardcoded BSB verse strings for the book-frame welcome screen, plus the version literal rendered between the book's bottom edges. Locked content per design D1 — the welcome screen is a pure view that must never fail on first run / no network. No JSDoc comments wrap the verses (R8/R9 don't apply, just a one-line `// Why:` header per house style).
- **Does NOT**: import anything from `src/domain/`, `src/application/`, or `src/api/`; call any port; perform any IO; format the text (the screen does layout). This is a content module, not a renderer.
- **Constants (full text, locked)**:
  - `GENESIS_1_1_TEXT = "In the beginning God created the heavens and the earth."` (BSB, Genesis 1:1)
  - `JOHN_3_16_TEXT = "For God so loved the world that He gave His one and only Son, that everyone who believes in Him shall not perish but have eternal life."` (BSB, John 3:16)
  - `WELCOME_VERSION = "v0.1.0"` — kept as a literal, NOT read from `package.json` at runtime. When `package.json` version bumps, this constant bumps in the same commit. The smoke test for the CLI path (`tests/smoke.test.ts`) does not depend on this string.

### NEW — `src/tui/welcome/welcome-reducer.ts`

- **Path**: `src/tui/welcome/welcome-reducer.ts`
- **Exports**: `WelcomeState`, `WelcomeAction`, `Effect`, `welcomeReducer`, `initialWelcomeState`
- **Purpose**: The pure state machine. Given the current `WelcomeState` and an incoming `WelcomeAction`, returns `[next state, Effect | null]`. Implements Rules 8/9/10 exactly: state and actions are discriminated unions, action names are past-tense facts, side effects are descriptor values returned to the driver (never invoked inline). The reducer is the file most directly mirrored by the future Bubble Tea `Update` function.
- **Behavioural contract**:
  - `("KeyPressed", "q")` → `[state, { kind: "quit" }]` (no state change; emit quit effect)
  - `("KeyPressed", "Q")` → same as `"q"` (case-insensitive convenience; tasks phase decides whether to also support `Ctrl+C`/`Esc`, but for this slice ONLY `q` and `Q` quit)
  - `("KeyPressed", any other)` → `[state, null]` (no-op; future slices add help/palette here)
- **Does NOT**: import any OpenTUI symbol; import React; touch `process`, `console`, or any IO; spawn any Promise; call any use case. Every test case is a single synchronous assertion on `welcomeReducer(state, action)`.

### NEW — `src/tui/welcome/welcome-screen.tsx`

- **Path**: `src/tui/welcome/welcome-screen.tsx`
- **Exports**: `WelcomeScreen`, `WelcomeScreenProps`
- **Purpose**: Pure JSX composition for the welcome view. Reads `BANNER` from `@/cli/banner`, `GENESIS_1_1_TEXT` / `JOHN_3_16_TEXT` / `WELCOME_VERSION` from `@/cli/welcome-content`. Composes `<box>` / `<text>` (or whatever `@opentui/react` primitives the apply phase confirms exist) into the locked book-frame layout. Receives `state` and `dispatch` as props from the driver; does NOT call `useReducer` or `useState` for business state.
- **Layout sketch (visual, not code)**:
  - Top region: `<text>` rendering `BANNER` verbatim (preserves whitespace).
  - Middle region: book-frame box. Two columns inside; left column = `GENESIS_1_1_TEXT` with `✦ Genesis 1:1` footer, right column = `JOHN_3_16_TEXT` with `✦ John 3:16` footer. The frame's angled edges (`╱` / `╲`) and drop-shadow (`░░`) come from the constant strings in `welcome-content.ts` OR from inline JSX — apply phase picks whichever is cleaner; the design's only requirement is "match `docs/ui-sketches.md` byte-for-byte."
  - Bottom region: `<text>` rendering `? help • q quit` (status hint, NOT inside the book frame — keeps verses uncluttered per `docs/ui-sketches.md`).
- **Does NOT**: own any state via `useState` for business meaning; call `useEffect` for any logic; import from `src/domain/`, `src/application/`, or `src/api/`; handle keypresses (the driver attaches the keypress listener and dispatches `KeyPressed` actions). The component is purely presentational.

### NEW — `src/tui/welcome/welcome-reducer.test.ts`

- **Path**: `src/tui/welcome/welcome-reducer.test.ts`
- **Exports**: none (test file)
- **Purpose**: Colocated unit tests for `welcomeReducer`. Uses `bun:test`. Imports only from `@/tui/welcome/welcome-reducer` — no OpenTUI, no React. Each test asserts one transition: `(state, action) → (state, effect)`.
- **Test cases** (apply phase implements exactly these):
  1. `welcomeReducer(initialWelcomeState, { type: "KeyPressed", key: "q" })` returns `[initialWelcomeState, { kind: "quit" }]`.
  2. `welcomeReducer(initialWelcomeState, { type: "KeyPressed", key: "Q" })` returns `[initialWelcomeState, { kind: "quit" }]`.
  3. `welcomeReducer(initialWelcomeState, { type: "KeyPressed", key: "x" })` returns `[initialWelcomeState, null]`.
  4. `initialWelcomeState.kind === "active"` (sanity check on the seed).
- **Does NOT**: spawn the renderer; call `tuiDriver`; render JSX; mock OpenTUI. The reducer is pure, so the test doesn't need any harness beyond `bun:test`.

> **Naming note**: the proposal listed this file as `welcome-reducer.test.ts`. The orchestrator brief mentioned `welcome-screen.test.ts`. We ship as `welcome-reducer.test.ts` because the file tests the reducer (not the JSX), which matches the `*.test.ts` colocation pattern used everywhere else in the codebase (`reference.test.ts`, `book-id.test.ts`, `get-passage.test.ts`).

### NEW — `src/tui/tui-driver.ts`

- **Path**: `src/tui/tui-driver.ts`
- **Exports**: `tuiDriver(): Promise<void>`
- **Purpose**: The TUI runtime entry. Initializes the OpenTUI renderer (`createCliRenderer` or whatever the installed version exposes — apply phase pins the exact import surface), mounts `<WelcomeScreen>` via `@opentui/react`'s `render`/`createRoot`, wires the keypress dispatch through `useReducer`, runs the effect loop, and resolves the returned Promise once the reducer emits `Effect.quit`. This is the **only** file in the codebase that holds an OpenTUI renderer handle.
- **Internal structure** (single file, ~50 lines):
  - `runEffect(effect, renderer, resolve)` — private helper: switches on `effect.kind`. For `"quit"`: calls `renderer.destroy()`, then calls `resolve()`. Exhaustive `never` fallthrough.
  - A small wrapper component `<WelcomeRoot>` (defined inline, not exported) that owns `useReducer(welcomeReducer, initialWelcomeState)` and watches for emitted effects via the dispatch interception pattern below.
  - The keypress listener (OpenTUI's `useKeyboard` or the imperative `renderer.on("keypress", ...)` API — apply phase confirms which is idiomatic in the installed version) calls `dispatch({ type: "KeyPressed", key })`.
  - **Effect dispatch pattern**: instead of running effects inside React via `useEffect` (Rule 9 violation), `<WelcomeRoot>` does NOT call `dispatch` directly. The driver wraps `dispatch` in a custom function that (a) calls the real reducer manually to get the effect, (b) updates state via the React dispatch, and (c) hands the effect to `runEffect`. Concrete shape inside the file:

  ```ts
  // sketch — apply phase finalises
  let resolveOuter: () => void;
  const done = new Promise<void>((r) => { resolveOuter = r; });

  function App() {
    const [state, baseDispatch] = useReducer(welcomeReducer, initialWelcomeState);
    // Sentinel: track the *last* action manually so we can read the
    // effect side-band. Reducer is called twice — once by React for
    // the state, once by us for the effect. Both calls are pure, so
    // this is safe (no IO).
    const dispatch = (action: WelcomeAction) => {
      const [, effect] = welcomeReducer(state, action);
      baseDispatch(action);
      if (effect !== null) {
        runEffect(effect, renderer, resolveOuter);
      }
    };
    return <WelcomeScreen state={state} dispatch={dispatch} />;
  }
  ```

  - **Why this shape over a `useEffect`-based effect runner**: Rule 9 explicitly forbids `useEffect` for business logic. Calling `welcomeReducer` twice is benign (pure function, no IO, microsecond cost) and keeps the effect runner outside React's render cycle. The Bubble Tea port doesn't have this problem (its `Update` returns `(model, cmd)` directly to the runtime), so this duplication is a TS-only ergonomic price for keeping the rule honest.
- **Does NOT**: parse argv; call `process.exit` (that's the entry point's job — D4); import from `src/domain/`, `src/application/`, or `src/api/`; render anything other than `<WelcomeScreen>`. Future TUI views will be mounted from this file when routing arrives, but for this slice it's welcome-only.

### NEW — `scripts/generate-banner.ts`

- **Path**: `scripts/generate-banner.ts` (NEW directory at repo root: `scripts/`)
- **Exports**: none (script entry)
- **Purpose**: One-shot dev utility. Runs synchronously, reads no inputs, writes `src/cli/banner.ts`. The script invokes `figlet` programmatically (via `import figlet from "figlet"`) — NOT `bunx`, because the proposal-locked `figlet` devDependency is already on disk after `bun install`, so calling it as a library is cheaper and more deterministic than spawning a subprocess. Output format: a TS file that exports `BANNER` as a single template literal.
- **Why a `scripts/` directory** (not `src/scripts/`): scripts are not application source — they're dev tooling. `src/` is reserved for code that ships in the compiled binary (`bun build --compile`). Putting `generate-banner.ts` outside `src/` makes the boundary obvious and keeps `tsconfig.json` `include` unchanged (`["src/**/*", "tests/**/*"]` does NOT include `scripts/**`, and the script is invoked via `bun run scripts/generate-banner.ts` which doesn't need tsconfig discovery).
- **Output shape** (the file the script writes):
  ```ts
  // src/cli/banner.ts — GENERATED by `bun run generate:banner`. Do NOT edit by hand.
  // Source: bunx figlet -f Isometric1 verbum (figlet v<version>).
  export const BANNER = `
        ___           ___           ___           ...
       /\\__\\         /\\  \\         /\\  \\         ...
      ...
  `;
  ```
  Note the leading newline inside the template literal — `figlet` output starts with a blank line that `<text>` renders as a top margin, matching `docs/ui-sketches.md`.
- **Regen workflow**:
  1. `bun install` (ensures figlet devDep is present)
  2. `bun run generate:banner`
  3. `git diff src/cli/banner.ts` (verify expected change)
  4. Commit `src/cli/banner.ts` together with whatever motivated the regen.
- **Does NOT**: get committed into the production bundle (`bun build --compile src/index.tsx` doesn't traverse `scripts/`); read or write any file outside `src/cli/banner.ts`; invoke network calls; modify `package.json`.

### MODIFIED — `src/index.tsx`

- **Path**: `src/index.tsx` (existing file)
- **Change**: Replace the existing "no args → stderr + exit 2" branch with the TUI dispatch.
- **New shape** (after the change):

  ```tsx
  // Why: the entry point stays a wiring layer — no business logic. A 3-line
  // argv branch routes to either the TUI (no args) or the existing CLI run().

  import { run } from "./cli/run";
  import { tuiDriver } from "./tui/tui-driver";

  const argv = Bun.argv.slice(2);
  ```

- **Unchanged**: the args-present path still calls `run(argv)` and exits with its returned code.

- **Verified safe** (per code reading): no JSX outside `src/index.tsx` (which is `.tsx` only by extension — it has no JSX) and the new `src/tui/**/*.tsx` files. `src/cli/run.ts`, `src/cli/render.ts`, `src/api/**/*`, `src/application/**/*`, `src/domain/**/*` are all `.ts` and contain no JSX. Flipping `jsxImportSource` is a no-op for those files.

  - Add `dependencies`: `@opentui/core`, `@opentui/react`, `react`. Versions pinned by `bun add` at apply time (whatever resolves to latest stable).

  - Add `scripts.generate:banner`: `"bun run scripts/generate-banner.ts"`.

## Banner build pipeline

How `bun run generate:banner` works, end to end:

1. **Trigger**: a contributor runs `bun run generate:banner` from the repo root.
2. **Resolution**: `package.json` `scripts.generate:banner` resolves to `bun run scripts/generate-banner.ts`.
3. **Execution**:
   - The script imports `figlet` (devDependency).
   - Calls `figlet.textSync("verbum", { font: "Isometric1" })` — synchronous because we want deterministic output. The figlet npm package ships its own font files; no system figlet binary needed.
   - Writes the result to `src/cli/banner.ts` using Bun's `Bun.write(...)`. The file body is the literal:
     ```ts
     // src/cli/banner.ts — GENERATED by `bun run generate:banner`. Do NOT edit by hand.
     // Source: figlet -f Isometric1 "verbum"
     export const BANNER = `<output>` ;
     ```
     where `<output>` is the figlet text inserted with backtick escaping (only `` ` `` and `${` need escaping, neither of which appear in figlet output for `verbum`).
4. **Verification**: contributor runs `git diff src/cli/banner.ts` to confirm the regenerated text matches expectation, then commits.
5. **Runtime**: the compiled binary imports `BANNER` as a static string. figlet is not present in `dependencies`, so it is NOT bundled. `bun build --compile` only traverses `src/`, which contains no figlet import.

**File format for `src/cli/banner.ts`** (locked):

```ts
// src/cli/banner.ts — GENERATED by `bun run generate:banner`. Do NOT edit by hand.
// Source: figlet -f Isometric1 "verbum"
export const BANNER = `
```

## Layer fit & house rules check

| Rule | How this design satisfies it |
|------|------------------------------|
| **Rule 7** — TUI business state in `useReducer`, not `useState`. | `<App>` (inline in `tui-driver.ts`) uses `useReducer(welcomeReducer, initialWelcomeState)`. No `useState` anywhere in this slice. |
| **Rule 8** — No `useEffect` for business logic. | The effect runner lives outside React. The `dispatch` wrapper in `<App>` calls `runEffect(effect, ...)` synchronously after `baseDispatch(action)`. No `useEffect` is used at all in this slice (it's not needed — there's no on-mount data fetch, no subscription teardown). The keypress listener is attached imperatively in `tui-driver.ts` against the renderer, not via `useEffect`. |
| **Rule 9** — Action names are past-tense facts. | `KeyPressed` is past-tense. The proposal's "user pressed q" → action `KeyPressed` mapping is exactly `house-rules.md`'s template. |
| **Rule 10** — No decorators. | None used. Composition is plain function calls (`runEffect(effect, renderer, resolve)`). |

**No conflicts**. The design satisfies rules 7–10 without contortion. Rule 1 (no domain throws) is unaffected because no domain code runs in this slice. Rule 2 (no classes) is unaffected — every new file uses functions and plain object literals. Rule 4 (Zod only in `src/api/`) is unaffected — no Zod here. Rule 5 (discriminated unions) is satisfied by `WelcomeState`, `WelcomeAction`, and `Effect`. Rule 6 (branded IDs) is unaffected. Rule 11 (no decorators) is unaffected. Rule 12 (`Promise<Result<T, E>>` for async data) is unaffected — `tuiDriver()` returns `Promise<void>` because it's not data-bearing; it's a runtime lifecycle Promise (analogous to `await app.listen()`).

## Bubble Tea portability mapping

Per ADR 0009, every TS construct in this slice MUST have a Go + Bubble Tea counterpart so the future port is mechanical transcription. Mapping:

| TS / OpenTUI symbol | File | Go + Bubble Tea counterpart | Notes |
|---|---|---|---|
| `BANNER: string` | `src/cli/banner.ts` | `const banner = "..."` in `internal/cli/banner.go` (or `//go:embed banner.txt`) | The figlet wordmark embeds as a string in both targets. No runtime renderer in either. |
| `GENESIS_1_1_TEXT`, `JOHN_3_16_TEXT`, `WELCOME_VERSION` | `src/cli/welcome-content.ts` | `const genesis11 = "..."` etc. in `internal/cli/welcome_content.go` | Plain string constants. 1:1. |
| `type WelcomeState = { kind: "active" }` | `welcome-reducer.ts` | `type welcomeState struct { kind string }` (or `type welcomeState string`) | Single-variant union → either Go enum string or single-field struct. |
| `type WelcomeAction = { type: "KeyPressed"; key: string }` | `welcome-reducer.ts` | `tea.KeyMsg` (built-in) | Bubble Tea ships `tea.KeyMsg` natively. Past-tense `KeyPressed` ports to `KeyPressedMsg` if a wrapper is wanted, but native `tea.KeyMsg` already carries `String()` for the key. Mechanical. |
| `type Effect = { kind: "quit" }` | `welcome-reducer.ts` | `tea.Cmd` (specifically `tea.Quit`) | Bubble Tea ships `tea.Quit` as a built-in command. Our descriptor maps directly. |
| `welcomeReducer(state, action) => [state, Effect | null]` | `welcome-reducer.ts` | `func (m welcomeModel) Update(msg tea.Msg) (tea.Model, tea.Cmd)` | The `[state, effect]` tuple is exactly Bubble Tea's `(Model, Cmd)` return shape. Mechanical. |
| `initialWelcomeState` | `welcome-reducer.ts` | `func initialModel() welcomeModel` | Bubble Tea convention is a constructor function instead of a const, but the content is identical. |
| `<WelcomeScreen state dispatch>` JSX tree | `welcome-screen.tsx` | `func (m welcomeModel) View() string` returning a multi-line string composed via `lipgloss.JoinVertical(...)` | OpenTUI's React tree becomes Bubble Tea's `View()` returning a string. Each JSX subtree (`<Banner/>`, book-frame, `<StatusLine/>`) becomes a helper string-builder. |
| `tuiDriver(): Promise<void>` | `tui-driver.ts` | `func runTUI() error` calling `tea.NewProgram(initialModel()).Run()` | The Promise-resolves-on-quit shape ports directly to Go's blocking `Program.Run()` returning when the program quits. |
| `runEffect(effect, renderer, resolve)` | `tui-driver.ts` (private) | Built into `tea.Program` (no equivalent helper needed) | Bubble Tea's program loop runs commands natively. The TS-side helper exists only because OpenTUI lacks a built-in effect dispatcher. The Go port deletes this helper. |
| `useReducer(welcomeReducer, initialWelcomeState)` (in `<App>`) | `tui-driver.ts` (inline `<App>`) | `tea.Program` model loop (no equivalent code; the runtime owns it) | React's reducer hook becomes the Bubble Tea program loop. The `dispatch` wrapper that intercepts effects is also unnecessary — `tea.Cmd` is returned directly from `Update`. |
| `argv` branch in `src/index.tsx` | `src/index.tsx` | `argv` branch in `func main()` | Identical structure: `if len(os.Args) == 1 { runTUI() } else { runCLI(os.Args[1:]) }`. |
| `renderer.destroy()` | `tui-driver.ts` (inside `runEffect`) | `tea.Program.Quit()` (or program returning naturally from `Update` after returning `tea.Quit`) | Bubble Tea handles terminal restoration in its program loop. Our manual `destroy()` is a TS-only quirk. |

**Bubble-Tea-mapping completeness**: every TS construct in this slice has a Go counterpart. Two TS-only ergonomic helpers (`runEffect` and the `dispatch` wrapper) have NO Go counterpart because Bubble Tea's runtime owns those concerns natively. Their absence in Go means the port is **simpler** in Go, not harder — the design "loses" complexity on port. That's the desired property.

## Test strategy

**What this slice tests** (one assertion per `(state, action) → [state, effect]` transition):

1. `welcomeReducer(initialWelcomeState, { type: "KeyPressed", key: "q" })` → `[initialWelcomeState, { kind: "quit" }]`
2. `welcomeReducer(initialWelcomeState, { type: "KeyPressed", key: "Q" })` → `[initialWelcomeState, { kind: "quit" }]`
3. `welcomeReducer(initialWelcomeState, { type: "KeyPressed", key: "x" })` → `[initialWelcomeState, null]`
4. `initialWelcomeState.kind === "active"` (sanity)

**What this slice does NOT test** (deferred to manual smoke):

- The actual rendered output of `<WelcomeScreen>` — would require mounting OpenTUI in a test harness, which is more work than it's worth for a static screen.
- Keypress event delivery from the OpenTUI renderer to the reducer — same reason.
- The `tuiDriver()` Promise lifecycle (initialise → render → keypress → effect → destroy → resolve) — requires a real terminal or a mock with raw-mode awareness.
- Process exit semantics (`process.exit(0)` after `await tuiDriver()`) — covered by manual smoke step in tasks phase.

**Manual smoke step** (lives in tasks phase, not a `bun test`):

> Run `bun run src/index.tsx` (no args). Confirm the welcome screen renders with the Isometric1 wordmark, the book-frame, and `? help • q quit` line. Press `q`. Confirm the shell prompt returns and the terminal is not garbled (cursor visible, normal mode restored).

If the manual smoke step fails because of Bun + OpenTUI lifecycle quirks, the documented fallback is to call `process.exit(0)` directly inside `runEffect`'s `quit` branch immediately after `renderer.destroy()`, bypassing the Promise resolution path. This is the "known-good escape hatch" the proposal called out.

**Existing tests must remain green**: `tests/smoke.test.ts` is unchanged. All 29 existing unit tests are unchanged. Apply phase verifies via `bun test`.

## Open questions

1. **Bun + OpenTUI process lifecycle on `q`** — the riskiest unknown surfaced in proposal and explore. Design's plan to retire it:
   - Spec the exact teardown sequence: `Effect.quit` → `renderer.destroy()` → `resolve()` → `await tuiDriver()` returns → `process.exit(0)`.
   - Document the manual smoke step in tasks phase.
   - Document the known-good escape hatch (call `process.exit(0)` directly inside `runEffect` if the resolve path doesn't unblock the await).
   - This is the only design item that cannot be resolved without running the code; design has done what design can do.

2. **OpenTUI primitive names** — `<box>` vs. `<group>` vs. `<view>`, attaching keypress via `useKeyboard` vs. `renderer.on("keypress", ...)`. The installed version of `@opentui/react` (whatever `bun add` resolves) determines the exact import surface. Apply phase pins the specific names; design specifies the **shape** ("a JSX element that renders the banner; a JSX element that renders the book-frame; a keypress hook that delivers `{ name: string }` events"). This is implementation friction, not architecture.

3. **`scripts/` outside `tsconfig.json` `include`** — confirmed safe: `bun run scripts/generate-banner.ts` does not require tsconfig discovery (Bun's runtime has its own loader). If type-checking the script becomes desirable, a future slice adds `scripts/**/*` to `tsconfig.include` — for now it's deliberately excluded so the production build's type surface stays focused on `src/` + `tests/`.

No other open questions. Locked decisions D1–D5 from the proposal are not re-litigated here; design specs the **how**.

## Architectural decisions log (ADR-style)

### ADR-DESIGN-WELCOME-1 — `null` for "no effect", not `Effect.None` variant

**Context:** The reducer needs a way to say "no side effect this dispatch."
**Decision:** Return `null` as the second element of the `[state, effect | null]` tuple.
**Rationale:** `house-rules.md` Rule 8's example uses `Effect | null`. The v1-architecture-spike never introduced an `Effect.None` variant convention, so this slice doesn't either. Adding `Effect.None` would force every future reducer to switch on a variant that means "do nothing" — wasted cycles. `null` is one fewer concept.
**Rejected alternatives:**
- *`Effect.None` discriminated-union variant*: more "pure" but YAGNI for this slice. Can be promoted later if a reducer ever needs to distinguish "no effect" from "absent effect" (e.g. for logging), but no current reducer does.
- *`undefined`*: avoid `undefined` in reducer signatures — TS treats `T | null` and `T | undefined` differently in nullability checks, and `null` is the explicit-absence convention this codebase already uses (see `parseArgs` defaults, `JSON.parse` outcomes).

### ADR-DESIGN-WELCOME-2 — Effect runner lives in `tui-driver.ts`, not React

**Context:** Where does the effect runner physically live? Inside `useEffect`? Inside the reducer? In a separate file?
**Decision:** A private `runEffect(effect, renderer, resolve)` helper inside `tui-driver.ts`. Called from the dispatch wrapper in `<App>`, NOT from `useEffect`.
**Rationale:** Rule 9 forbids `useEffect` for business logic. The driver is the right physical home because it's the only file that holds the renderer handle and the outer Promise resolver. Putting `runEffect` in its own file would require exporting `renderer` (architectural smell) or threading both `renderer` and `resolve` through a public API (overkill for one effect).
**Rejected alternatives:**
- *Effect runner inside `<App>`'s `useEffect(() => { /* run effect when state changes */ }, [state])`*: Rule 9 violation. Also requires a "last effect" piece of state to trigger the hook, which complicates the model.
- *Effect runner as an exported function in `src/tui/effect-runner.ts`*: file proliferation for one switch case. Promotes when a second effect type appears (e.g. `save_preferences` in a future slice).
- *Reducer invokes the effect inline*: Rule 9's exact prohibition. Couples the reducer to the renderer, breaks pure-function tests.

### ADR-DESIGN-WELCOME-3 — `welcome-content.ts` and `banner.ts` live in `src/cli/`, not `src/tui/`

**Context:** The verses and the wordmark are used by the welcome TUI screen. Should they live with the TUI?
**Decision:** Both files live in `src/cli/` (`src/cli/banner.ts`, `src/cli/welcome-content.ts`).
**Rationale:** `docs/ui-sketches.md` shows the same banner reused by `verbum --version` (a future CLI flag). Putting it under `src/tui/` would force the future CLI dispatcher to import from `src/tui/`, inverting the layering (CLI is older / lower-coupling than TUI in this codebase). `src/cli/` is the natural home for "presentation constants used by either driver."
**Rejected alternatives:**
- *`src/tui/welcome/banner.ts`*: forces `--version` flag handling (a CLI concern) to import from `src/tui/`.
- *`src/shared/`*: no other shared layer exists yet. Premature.
- *Top-level `assets/`*: file is TS, not a binary asset. Doesn't fit.

### ADR-DESIGN-WELCOME-4 — `tuiDriver` returns `Promise<void>`, entry point owns `process.exit`

**Context:** Where does `process.exit(0)` live after a clean TUI quit?
**Decision:** `tuiDriver(): Promise<void>` resolves on `Effect.quit` (after `renderer.destroy()`). `src/index.tsx` does `await tuiDriver(); process.exit(0)`.
**Rationale:** Symmetric with the existing CLI path, where `run()` returns a number and the entry point exits. Keeps the driver pure (no `process.exit` inside the driver — testable in principle even if we don't write the test). The explicit `process.exit(0)` at the entry is belt-and-braces against any lingering Bun handles that would block natural termination.
**Rejected alternatives:**
- *`tuiDriver()` calls `process.exit(0)` directly*: untestable, asymmetric with `run()`. Reserved as the documented escape hatch if the await-then-exit path fails on Bun.
- *`tuiDriver()` returns an exit code (number)*: only one exit code (`0`) is reachable from the welcome screen, so the `Promise<number>` shape would always resolve to `0`. Useless type complexity.

### ADR-DESIGN-WELCOME-5 — `scripts/` directory, not `src/scripts/`

**Context:** Where does `generate-banner.ts` live?
**Decision:** `scripts/generate-banner.ts` (new top-level directory).
**Rationale:** Scripts are dev tooling, not application source. `src/` is reserved for code shipped in the compiled binary. Putting `generate-banner.ts` outside `src/` keeps `tsconfig.json` `include` unchanged and makes the boundary obvious to future contributors. Matches widespread convention in JS/TS projects.
**Rejected alternatives:**
- *`src/scripts/`*: pollutes the production source tree.
- *`tools/` or `bin/`*: less idiomatic in JS/TS land than `scripts/`.
- *Inline in `package.json`*: figlet usage is multi-line; embedding it in a `scripts.generate:banner` shell snippet is brittle.

### ADR-DESIGN-WELCOME-6 — Reducer called twice per dispatch (once by React, once by the wrapper)

**Context:** OpenTUI's `useReducer` only returns `[state, dispatch]`, no effect channel. How does the runtime get the `Effect` from a transition?
**Decision:** The dispatch wrapper in `<App>` calls `welcomeReducer(state, action)` manually before calling `baseDispatch(action)`. The first call extracts the effect; the second updates React state.
**Rationale:** Calling a pure function twice is benign (microsecond cost, no IO, no allocations beyond a tuple). It avoids `useEffect` (Rule 9). It avoids a side-band ref/state to thread effects through React. It keeps the reducer signature `(state, action) => [state, effect]` — the exact Bubble Tea shape.
**Rejected alternatives:**
- *Use a `useRef` to capture the last effect inside the reducer's closure*: requires the reducer to be defined inside the component (closure over the ref), losing referential equality and breaking the "reducer is pure" property. Also forces a `useEffect` to consume the ref. Rule 9 violation.
- *Wrap the reducer to return only `state`, store effects in a module-level queue*: module-level mutable state is a Go-portability landmine (no clean Bubble Tea equivalent).
- *Use a third-party library (Redux, Zustand) with built-in effect middleware*: adds a runtime dep for one effect. Massive overkill.

The duplicate-call cost is a TS-only quirk. The Bubble Tea port deletes this entirely (Bubble Tea's runtime consumes `(Model, Cmd)` directly).
